从 0 到上线：一个小白是如何把博客搭起来的#

为什么我要一个博客#

在现在这个信息过载的环境里，你几乎无法避免被各种观点包围。

问题不在于信息多，而在于—— 绝大多数信息是情绪化的、碎片化的、低质量的。

它们不断出现，不断强化，很容易让人产生一种错觉： 你在“了解世界”，但实际上，你只是被不断塑造。

我开始意识到，如果没有一个稳定的输出空间， 我的想法只会停留在当下，很快被新的信息覆盖。

所以我想要一个地方：

  • 不是用来表达情绪
  • 而是用来记录判断
  • 不是追热点
  • 而是形成长期思考

博客对我来说，不是工具，而是一种约束： 逼迫自己把模糊的想法整理成清晰的结构。

为什么不用现成平台#

最直接的原因其实很简单：不喜欢。

大多数平台都在做同一件事—— 尽可能延长你的停留时间。

于是就有了：

  • 推荐算法
  • 广告
  • 各种功能堆叠

它们的目标不是让你思考，而是让你“停下来”。

但我想要的是一个完全相反的东西：

  • 没有干扰
  • 没有推送
  • 没有任何试图影响我表达的机制

除此之外，还有一个更现实的原因：

我不信任第三方平台对内容的长期承载能力。

内容可能被删、被限流、被埋没， 而你几乎没有任何控制权。

所以我选择了一条更麻烦的路：

用 Hugo 搭建一个静态博客，托管在 GitHub 上。

它很简单，但也很“干净”。

搭建过程：看似简单，实际全是坑#

这套方案在教程里通常被描述为“新手友好”。

但实际体验是：

每一步都可能卡住。

我遇到的几个典型问题，本质上并不是“操作错误”，而是理解缺失。

1. 工具链不熟（Git / SSH）#

最开始连主题都装不上。

原因不是步骤错了，而是：

我并不理解 Git 在做什么。

当你只是“照着命令敲”的时候，一旦出错，你就完全失去判断能力。

2. 网络与环境问题（GitHub 连接）#

在国内环境下，GitHub 的连接并不稳定：

  • 超时
  • 连接失败
  • SSH / HTTPS 混乱

这些问题不会告诉你“该怎么做”， 只会给你一堆看不懂的报错。

3. 子模块与部署机制（最坑的一点）#

最难的不是写代码，而是理解：

为什么本地可以，线上却不行？

比如：

  • 主题没有正确拉取
  • Actions 构建成功，但页面是 404
  • 实际上部署的是“空站”

这些问题的共同点是：

👉 你以为你做对了，但系统并没有按你理解的方式运行

我是怎么解决这些问题的#

一开始的状态是：乱试。

不停地改配置、重跑、刷新。

但很快发现，这样是无效的。

后来我逐渐改变了方式：

1. 把报错当作信息，而不是障碍#

报错不是“失败”，而是：

👉 系统在告诉你它哪里没按预期工作

当你开始认真读报错，而不是忽略它， 问题就开始变得“可定位”。

2. 从“跟教程”转向“理解结构”#

教程告诉你“怎么做”， 但不会告诉你“为什么”。

只有当你理解了：

  • Hugo 是如何生成页面的
  • GitHub Actions 在做什么
  • submodule 是什么

你才真正具备解决问题的能力。

3. 接受“慢”的过程#

这件事没有捷径。

你不可能在完全不理解的情况下，一次成功。

但每解决一个问题，你对系统的理解就多一层。

最后的结果#

博客上线了。

但这件事本身，其实没有那么重要。

更重要的是：

我第一次完整地经历了一个过程：

  • 从完全不懂
  • 到不断出错
  • 再到逐渐理解
  • 最终解决问题

一点总结#

这次经历给我最直接的几个感受是：

  1. 报错不是问题，忽略报错才是问题
  2. 技术问题的本质，往往是理解问题
  3. 搭建博客这件事，本质上是在训练解决问题的能力

博客只是一个开始。

真正有价值的，是你在这个过程中建立起来的东西。
